# El modelo de zonas horarias de `pytz`

<div class="bullets-with-header">

El objecto `timezone` es una fabrica que adjunta el `tzinfo`:

<div class="small-spacer"></div>

```pycon
>>> LOS_p = pytz.timezone('America/Los_Angeles')
>>> dt = LOS_p.localize(datetime(2017, 8, 11, 14, 0))
>>> print_tzinfo(dt)

2017-08-11 14:00:00-0700
    tzname:   PDT;      UTC Offset:  -7.00h;        DST:      1.0h
```


</div>
<div class="bullets-with-header">

Los `tzinfos` son todos *desplazamientos fijos*:
<div class="small-spacer"></div>

```pycon
>>> print(repr(LOS_p))
&lt;DstTzInfo 'America/Los_Angeles' LMT-1 day, 16:07:00 STD>

>>> print(repr(dt.tzinfo))
&lt;DstTzInfo 'America/Los_Angeles' EDT-1 day, 20:00:00 DST>
```

</div>

<div class="bullets-with-header fragment highlight-dashed">

Puedes escoger el lado de una transición cuando llamas a `localize`:

```pycon
>>> ambiguo = datetime(2004, 10, 31, 1, 30)
>>> print_tzinfo(LOS_p.localize(ambiguo, is_dst=False))
2004-10-31 01:30:00-0800
tzname:   PST;      UTC Offset:  -8.00h;        DST:      0.0h
```

</div>

El modelo de Python está diseñado ser *lazy* (perezoso), pero lo de `pytz` es *eager* (ansioso).

</div>

<div class="small-spacer"></div>

Notes:

Bueno, quizás te estés diciendo: "¿No es cierto que `pytz` existió durante diez años y todo el mundo lo usó con mucho éxito?". ¿Cómo resolvió `pytz` este problema? ¿Podemos hacerlo de la misma manera?

Y eres muy listo, querido oyente hipotético, porque sí, `pytz` sí que lo había resuelto, pero la verdad es que el modelo de zonas horarias que tiene `pytz` es fundamentalmente diferente de lo de `datetime`.

El problema es que no hay suficiente información guardada en el objeto `datetime` para disambiguar entre dos `datetime`s diferenciados por su desplazamiento, así que cuando creas el `datetime`, `pytz` crea un `tzinfo` **fijo** de manera *eager*, y guarda la información ahí dentro.

➡️ Y para escoger qué lado de un cambio quieres, puedes usar el parámetro `is_dst` para especificar el lado de hora de verano o el de hora estándar.

# Manejando horas ambiguas e imaginarias con `pytz`

Ya que se evalúan los desplazamientos de manera ansiosa, es posible representar `datetime`s que solo se diferencian por su desplazamiento por ajuntandoles desplazamientos diferentes a ellos.

El argumento `is_dst` de los métodos fabrica tiene tres modos:

<div class="is-dst-grid">

<div class="grid-left">
<span class="value"><code><span class="param_name">is_dst</span>=<span class="value">False</span></code></span><br/>
elige el lado STD si ambiguo<br/>
<em>(por defecto)</em>
</div>

<div>

```python
>>> ambiguo = datetime(2004, 10, 31, 1, 30)
>>> print_tzinfo(LOS_p.localize(ambiguo, is_dst=False))
2004-10-31 01:30:00-0500
tzname:   EST;      UTC Offset:  -5.00h;        DST:      0.0h
```

</div>

<div class="grid-left">

`is_dst=True`<br/>
elige el lado DST si ambiguo
</div>

<div>

```python
>>> print_tzinfo(LOS_p.localize(ambiguo, is_dst=True))
2004-10-31 01:30:00-0400
tzname:   EDT;      UTC Offset:  -4.00h;        DST:      1.0h
```

</div>

<div class="grid-left">
<code>is_dst=None</code><br/>
Eleva un error si ambiguo o imaginario
</div>

<div>

```python
>>> LOS_p.localize(ambiguo, is_dst=None)
AmbiguousTimeError: 2004-10-31 01:30:00
```

</div>

<div class="small-spacer"></div>

</div>

</div>

Notes:

Y esto resuelve el problema porque permite que `pytz` acepte un argumento para seleccionar una estrategia para resolver horas ambiguas e imaginarias. Puedes usar `is_dst` para indicar qué lado del cambio quieres, o si debería lanzar un error en casos ambiguos o imaginarios.

# Problemas con el modelo de `pytz`

* Requiere calculación ansiosa — ajuntar una zona `pytz` directamente da resultos equivocados:

  ```
  >>> dt = datetime(2020, 5, 1, tzinfo=LOS_p)
  >>> print_tzinfo(dt)
  2020-05-01 00:00:00-0753
      tzname:   LMT;      UTC Offset:  -7.88h;        DST:      0.0h
  ```

* Tienes que `normalize()` datetimes después de hacer aritmética con ellos:
 
  ```python
  >>> dt = LOS_p.localize(datetime(2020, 5, 1))
  >>> dt_add = dt + timedelta(days=180)
 
  >>> print_tzinfo(dt_add)
  2018-02-07 14:00:00-0700
      tzname:   PDT;      UTC Offset:  -7.00h;        DST:      1.0h
 
  >>> print_tzinfo(LOS_p.normalize(dt_add))
  2018-02-07 13:00:00-0800
      tzname:   PST;      UTC Offset:  -8.00h;        DST:      0.0h
  ```

</div>
</div>

Notes:

Pero no elegimos este modelo en la biblioteca estándar, porque tiene problemas.

Primero, tienes que llamar a `localize`. Si pasas un objeto de `pytz` directamente al constructor de `datetime`, va a darte algo extraño como esta chorrada con un desplazamiento de minutos no enteros.

Y poca gente se da cuenta de eso y acaban teniendo errores en su código.

Igualmente, si haces aritmética con ellos, el comportamiento de Python es hacer los cálculos a la parte naíf y después añadir el `tzinfo` original, pero los `tzinfo`s de `pytz` son zonas fijas, así que si llegas más allá de un cambio de reloj, tienes que llamar al método `normalize()` para recalcular cuál desplazamiento aplica.

Y una vez más, poca gente sabe que tienes que hacer esto y eso causa errores.

Así que la ergonomía no es lo suficientemente buena para ser la solución.
