# `pytz`'s time zone model
<br/>

* `tzinfo` is attached *by the time zone object itself*:

```python
>>> LOS_p = pytz.timezone('America/Los_Angeles')
>>> dt = LOS_p.localize(datetime(2017, 8, 11, 14, 0))
>>> print_tzinfo(dt)


2017-08-11 14:00:00-0700
    tzname:   PDT;      UTC Offset:  -7.00h;        DST:      1.0h
```
<br/>

* `tzinfo`s are all *static offsets*:

```python
>>> print(repr(LOS_p))
&lt;DstTzInfo 'America/Los_Angeles' LMT-1 day, 16:07:00 STD>

>>> print(repr(dt.tzinfo))
&lt;DstTzInfo 'America/Los_Angeles' EDT-1 day, 20:00:00 DST>
```

<br/>

* Python's model is designed to be lazy, but `pytz`'s model is *eager*

Notes:

So you might be saying to yourself, "Well, didn't `pytz` exist for like 10 years and everyone used it very profitably? How did `pytz` solve this? Maybe we can solve it the same way."

`pytz` did have a solution, but its time zone model is actually fundamentally different than `datetime`'s. Instead of providing objects that embody a set of rules and lazily calculating offsets, `pytz` time zones are basically factories for datetimes. You create a time zone object and call the `.localize()` method on it, passing in a naive datetime. It then eagerly calculates the offset, name, and DST status for that specific point in time, and attaches a *fixed offset* zone to the result. Essentially, `pytz` gives you a separate static offset object for each datetime you localize.

--

# Handling ambiguous and imaginary times in `pytz`

Because offsets are eagerly evaluated, it is possible to represent `datetime`s that differ only in their offset by attaching different offsets to them. `pytz`'s `is_dst` has three modes:

<br/>

- `is_dst=False` (default): choose the STD side if ambiguous

```python
>>> ambiguous = datetime(2004, 10, 31, 1, 30)

>>> print_tzinfo(LOS_p.localize(ambiguous, is_dst=False))
2004-10-31 01:30:00-0500
    tzname:   EST;      UTC Offset:  -5.00h;        DST:      0.0h
```
<br/>

- `is_dst=True`: choose the DST side if ambiguous

```python
>>> print_tzinfo(LOS_p.localize(ambiguous, is_dst=True))
2004-10-31 01:30:00-0400
    tzname:   EDT;      UTC Offset:  -4.00h;        DST:      1.0h
```
<br/>

- `is_dst=None`: Throw an error if ambiguous

```python
>>> LOS_p.localize(ambiguous, is_dst=None)
AmbiguousTimeError                        Traceback (most recent call last)
...
    362         if is_dst is None:
--> 363             raise AmbiguousTimeError(dt)

AmbiguousTimeError: 2004-10-31 01:30:00
```

Notes:

This eager model allows `pytz` to handle ambiguous and imaginary times by taking an extra parameter, `is_dst`. If you pass `is_dst=False` and the time is ambiguous, it chooses the standard time side; if you pass `is_dst=True`, it chooses the daylight saving time side. You can even pass `is_dst=None` to have it throw an error if the time is ambiguous. It's able to differentiate between these because it simply returns a different static offset object depending on your choice.

# Problems with `pytz`'s time zone model
<br/>

* Requires eager calculation — directly attaching a `pytz` timezone gives the wrong results:

  ```
  >>> dt = datetime(2020, 5, 1, tzinfo=LOS_p)
  >>> print_tzinfo(dt)
  2020-05-01 00:00:00-0753
      tzname:   LMT;      UTC Offset:  -7.88h;        DST:      0.0h
  ```
  <br/>

* You must `normalize()` datetimes after you've done some arithmetic on them:
  <br/>
 
  ```python
  >>> dt = LOS_p.localize(datetime(2020, 5, 1))
  >>> dt_add = dt + timedelta(days=180)
 
  >>> print_tzinfo(dt_add)
  2018-02-07 14:00:00-0700
      tzname:   PDT;      UTC Offset:  -7.00h;        DST:      1.0h
 
  >>> print_tzinfo(LOS_p.normalize(dt_add))
  2018-02-07 13:00:00-0800
      tzname:   PST;      UTC Offset:  -8.00h;        DST:      0.0h
  ```

Notes:

But there are significant problems with this model. For one thing, you actually *need* to do that localization step. if you directly attach one of these time zones to your datetime in the constructor, it's going to give you a weird, wrong answer — usually the "Local Mean Time" (LMT), which is some non-integer number of hours away from UTC.

Similarly, if you do datetime arithmetic — like adding 180 days to move past a DST transition — `datetime` will simply attach the same static `tzinfo` object to the output. Since that object is just a fixed offset, it won't reflect the new rules. To fix this, you have to remember to call the `.normalize()` method to trigger a recalculation. A lot of people don't realize this, which leads to many subtle bugs.
