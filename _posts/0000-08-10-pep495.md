# PEP 495: Local Time Disambiguation

* First introduced in Python 3.6 to solve the ambiguous time problem
* Introduces the `fold` attribute of `datetime`
* Changes to aware datetime comparison around ambiguous times

<br/>
<br/>

Whether you are on the fold side is a *property of the datetime*:

<br/>

```python
>>> print_tzinfo(datetime(2004, 10, 31, 1, 30, tzinfo=NYC))          # fold=0
2004-10-31 01:30:00-0400
    tzname:   EDT;      UTC Offset:  -4.00h;        DST:      1.0h

>>> print_tzinfo(datetime(2004, 10, 31, 1, 30, fold=1, tzinfo=NYC))
2004-10-31 01:30:00-0500
    tzname:   EST;      UTC Offset:  -5.00h;        DST:      0.0h
```

<br/>

**N.B.:** `fold=1` represents the *second* instance of an ambiguous datetime.

Notes:

The ergonomics of needing to manually call `.normalize()` were not great and led to many bugs, so that wasn't going to be the long-term solution. The solution that was introduced in Python 3.6 was to add an extra bit of information to the naive portion of the datetime: the `fold` attribute.

The idea is that you use this attribute to specify which side of an ambiguous transition you want to be on. `tzinfo` objects can then read the `fold` value and return different offsets accordingly. By convention, `fold=0` represents the first instance of the wall time, and `fold=1` represents the second instance.

Notes:

This is actually a bit different from how `pytz` handles things with its `is_dst` parameter. `pytz` asks you to select whether you want the DST side or the STD side of a transition. PEP 495's `fold` attribute instead asks whether you want the offset that applied *before* the transition or the one that applied *after*.

This turns out to be easier to implement and more well-defined because, as we learned earlier, some transitions don't actually change the DST status — they might be a change in the base offset where you have standard time on both sides. In those cases, "is DST" is ambiguous, whereas "before or after the fold" remains clear.
